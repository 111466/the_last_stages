# Tiny Swords 塔防项目开发进度与迭代计划

更新日期：2026-05-05
当前完成迭代：A 和 B 已完成

---

## 一、当前版本定位

当前项目已经完成一个 `经典固定路线塔防` 的核心验证版，具备基础战斗循环、波次管理、塔与敌人的基础交互，以及可运行的 HUD 与状态切换逻辑。

从本轮开始，项目的后续迭代方向正式调整为：

- 从“固定路线塔防”升级为“`地形争夺 / 动态路线塔防`”
- 核心卖点改为：`敌人会改路，玩家也能塑造路线`
- 核心体验改为：`双方围绕地图通路与瓶颈点进行持续争夺`

这意味着现有版本不再只是最终产品雏形，而是新的动态路线系统的底座原型。

## 二、当前已完成内容

### 1. 已完成的模块拆分

当前 `scripts/` 目录已经具备以下基础模块：

**原有模块：**
- `main.lua`
  - 负责游戏状态管理、输入处理、主循环更新、渲染调度
- `Config.lua`
  - 保存世界尺寸、初始金币/生命、建造点、基础颜色配置
- `Utils.lua`
  - 通用数学和坐标转换工具

**新增动态路线模块（迭代A/B完成）：**

- `RoutePlanner.lua`
  - A*寻路算法实现、路径计算、路径可视化
- `Structure.lua`
  - 管理路障/工事对象、生命值、受击反馈
- `BuildValidator.lua`

## 三、新方向的核心设计目标

本项目后续的核心设计目标，不再是做一个内容更多的普通塔防，而是做一个围绕 `路线控制权` 展开的动态攻防塔防。

### 1. 核心卖点

- 敌人不再只是沿着预设路线前进，而是会主动 `拆墙、爆破、开路`
- 玩家不再只是放塔，而是可以通过 `路障、墙体、壕沟、开关门` 改造战场
- 路线会在战斗中持续变化，同一关卡不再只有一种标准解

### 2. 核心体验

- 玩家要主动设计“让敌人去死的路线”
- 敌人会试图拆掉玩家设计好的火力走廊
- 每一波战斗的关键问题不只是“火力够不够”，还包括“路线还能不能守住”

### 3. 首版不追求的内容

为了控制复杂度，首版动态路线系统暂不追求以下目标：

- 完整自由地形编辑
- 敌人无限制地随意改路
- 同时上线拆墙、搭桥、架梯、钻地四大系统
- 大量新塔种和复杂分支养成

首版重点只验证两件事：

1. `玩家控路是否有趣`
2. `敌人破路是否能形成有效博弈`

## 四、当前交互说明与阶段性取舍

### 1. 现阶段已支持的操作

- `Enter` / 点击屏幕：开始游戏
- `1` / `2` / `3`：切换当前准备放置的塔类型
- 左键点击建造点：放置塔
- 左键点击已建造的塔：选中塔
- `U`：升级当前选中的塔
- `P`：暂停 / 继续
- `R`：重新开始
- `Esc`：返回开始界面

### 2. 当前版本的限制

为了优先跑通原型，目前仍保留以下简化设计：

- 建塔方式仍为固定建造点，不是自由网格放置
- 地图路径仍使用手写路径点，尚未切换到网格寻路
- 还没有 `可破坏路障`、`工兵敌人`、`路径重算`
- 目前没有正式特效、音效和精灵动画
- 没有对象池优化，敌人和投射物仍是直接创建和销毁
- 当前参数还未进入正式数值平衡

### 3. 接下来必须改变的设计点

如果要支持动态路线系统，以下设计必须逐步替换：

- `固定路径点` 要升级为 `网格/节点式路线系统`
- `固定建造点` 要升级为 `可控工事与限定建造区并存`
- `敌人只会前进` 要升级为 `敌人具备施工 / 破坏 / 绕路评估能力`
- `Path.lua` 的职责要被拆分为更细的导航与地形模块

## 五、当前已知问题与风险

### 1. 必须尽快验证的基础问题

- 需要在 UrhoX 运行环境中实际测试鼠标点击和触屏事件是否都正常
- 需要确认 `urhox-libs/UI` 在当前工程中对绝对定位和遮罩显示是否稳定
- 需要确认 NanoVG 绘制层与 UI 层在不同分辨率下是否都正常

### 2. 动态路线方向新增的核心风险

- 路线频繁变化会带来更高的寻路和状态同步成本
- 如果敌人改路太自由，玩家会感觉布局被随意否定
- 如果玩家封路过强，可能出现关卡被完全锁死的问题
- 如果地图反馈不清晰，玩家会看不懂哪段路可走、哪段路正在被破坏
- 如果前期就叠太多机制，教学成本会明显上升

### 3. 当前逻辑层面的不足

- 路径和建造点还没有与 `assets/map.json` 地图真正对齐
- 没有塔出售、暂停菜单、倍速、下一波按钮等操作功能
- 没有工事建筑、路线阻断、反施工单位
- 没有敌人特殊施工技能、塔分支升级、Boss 机制
- 还没有完整的经济节奏设计
- 暂时没有测试脚本和自动化验证

### 4. 美术与表现上的注意点

- 需要尽早明确 `墙体 / 路障 / 壕沟 / 城门 / 桥口` 的视觉语言
- 需要用明显反馈区分 `可走`、`被封堵`、`施工中`、`被破坏`
- 替换美术时应尽量只改渲染层，不改玩法逻辑层

## 六、后续迭代总目标

新的总目标建议按以下顺序推进：

1. 先稳住当前固定路线版本的运行与输入稳定性
2. 再把地图表示方式升级为可支持 `动态通路计算` 的结构
3. 然后先实现 `玩家控路`
4. 再实现 `敌人破路`
5. 最后补内容、表现、平衡与移动端适配

换句话说，项目后续不再是“先贴图再加内容”，而是：

`先验证动态路线的可玩性，再进入正式内容生产`

目标：先让玩家拥有基础的“地形争夺权”

- `Structure.lua`
- `BuildValidator.lua`
- `RouteDebug.lua`

完成标志：

- 玩家可以通过放置工事改变敌人的行进路线
- 玩家无法通过非法封路直接破坏关卡
- 同一张图至少可以导出两种以上有效火力走廊

### 迭代 D：敌人破路 MVP

目标：让敌人具备针对玩家布局的基础反制能力

建议任务：

- 新增 `工兵` 敌人，优先攻击玩家工事
- 新增 `爆破兵` 或精英敌人，用于打开关键缺口
- 定义施工目标选择规则，例如优先最近瓶颈或最短改道收益点
- 加入施工中状态、预警表现和工事受损反馈
- 为波次系统增加施工单位配置字段

建议新增模块：

- `ConstructionAI.lua`
- `ThreatEvaluator.lua`
- `StructureDamage.lua`

完成标志：

- 敌人不再只会走最短路，而是会尝试破坏玩家控路结果
- 玩家能够识别并优先处理工兵威胁
- 路线变化能形成明确的战术压力，而不是随机扰动

### 迭代 E：动态路线扩展与内容成型

目标：在拆墙破路成立后，逐步增加更高阶的路线争夺内容

建议任务：

- 根据地图类型选择扩展 `搭桥` 或 `架梯` 其中一种机制
- 增加反施工塔或反施工技能
- 增加更明确的地图关键节点，例如桥口、城门、壕沟、高地边缘
- 重新设计敌人波次结构，使其包含“破路编组”
- 加入至少 1 个会改变战场结构的 Boss

完成标志：

- 动态路线不再只是单一堵路玩法，而是具备多层次对抗
- 至少完成 1 张能体现路线争夺特色的主打关卡

### 迭代 F：正式美术与战斗表现

目标：把动态路线系统从逻辑原型推进为可展示版本

建议任务：

完成标志：

目标：保证动态路线玩法在完整关卡中稳定、清晰且可持续扩展
