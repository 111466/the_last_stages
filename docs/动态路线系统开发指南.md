# Tiny Swords 动态路线系统开发指南

更新日期：2026-05-05

---

## 一、文档目的

本指南用于指导项目从当前的 `固定路线塔防原型` 演进为 `地形争夺 / 动态路线塔防`。

这份文档重点回答以下问题：

- 动态路线系统到底要做什么
- 首版应该保留哪些规则，砍掉哪些复杂内容
- 需要新增哪些模块
- 开发顺序应该如何安排
- 如何验证系统是否真的有趣

本指南默认基于当前工程的 `UrhoX + Lua + NanoVG` 架构编写。

---

## 二、系统目标

### 1. 核心目标

动态路线系统的目标不是让敌人随机乱走，而是让 `地图通路本身` 成为战斗对象。

在这个系统里：

- 玩家可以通过工事改变敌人路线
- 敌人可以通过施工或破坏反制玩家布局
- 路线会在战斗中动态变化
- 战斗的关键不只是火力，而是 `路线控制权`

### 2. 玩家体验目标

理想体验应该是：

- 玩家主动设计一条“火力走廊”
- 玩家观察敌人是否试图绕路或破口
- 玩家在关键节点进行封锁、补防、抢修或放弃
- 敌我双方围绕桥口、门口、瓶颈位持续博弈

### 3. 系统成功标准

如果系统设计成功，玩家应该能明确感受到：

- 我在“塑造战场”，而不是只是在摆塔
- 敌人会试图拆掉我的计划
- 同一张图可以打出不同防线结构
- 我失败不是因为纯数值不够，而是因为路线争夺失败

---

## 三、首版范围控制

### 1. 首版必须有的内容

- 地图具备 `可通行 / 阻挡 / 可施工 / 可破坏` 基础状态
- 玩家可以建造至少一种工事，例如 `路障` 或 `木墙`
- 工事变化会触发敌人路径重算
- 敌人中至少有一种能针对工事的单位，例如 `工兵`
- 系统能阻止玩家把敌人彻底封死

### 2. 首版暂时不要做的内容

- 全地图自由编辑
- 复杂地形模拟
- 同时实现拆墙、搭桥、架梯、钻地全部能力
- 太多塔种与分支升级
- 高复杂度资源链与经营系统

### 3. 推荐的首版定义

首版建议只做：

- 玩家：`放置路障`
- 敌人：`工兵拆除路障`
- 地图：`网格通行 + 路径重算`
- UI：`路线预警 + 非法封路提示`

这是最小但完整的动态路线闭环。

## 四、系统规则总览

### 1. 地图规则

地图应至少包含以下地块类型：

- `road`
  - 默认道路，寻路成本最低
- `ground`
  - 可通行普通地面，成本略高
- `block`
  - 默认不可通行，例如岩石、建筑、深水
- `buildable`
  - 允许玩家建塔或建工事的区域
- `fortifiable`
  - 允许玩家放置路障/墙体的关键地块
- `destructible`
  - 可被敌人拆除或破坏的工事地块
- `goal`
  - 终点
- `spawn`
  - 出生点

实际实现时，一个地块可以同时拥有多个标签，例如：

- `buildable + fortifiable`
- `road + destructible`

### 2. 路线规则

敌人的移动不是基于写死路径点，而是基于当前地图状态实时计算。

基础原则：

- 系统始终尝试为敌人找到从 `spawn` 到 `goal` 的最低成本路径
- 工事、阻挡、危险地块会改变路径成本
- 当地图状态变化时，允许触发路径重算

### 3. 路径成本建议

建议先使用简单成本表：

| 地块 | 是否可通行 | 基础成本 |
|------|------------|----------|
| `road` | 是 | 1 |
| `ground` | 是 | 2 |
| `slow_zone` | 是 | 3 |
| `hazard_zone` | 是 | 4 |
| `wall` | 否 | 无 |
| `ditch` | 否 | 无 |
| `gate_open` | 是 | 1 |
| `gate_closed` | 否 | 无 |

这样做的好处是：

- 易调试
- 易观察
- 后续可逐步扩展到更复杂的权重系统

### 4. 动态改路的触发时机

以下行为会触发路径重算：

- 玩家放置工事
- 玩家拆除工事
- 敌人破坏工事
- 地图机关开关状态改变
- 特殊敌人临时改变地块状态

建议不要每帧都全图重算，而是采用：

- 事件触发重算
- 关键区域局部重算
- 路线缓存失效后再刷新

## 五、敌我双方的核心动作

### 1. 玩家动作

玩家在动态路线系统中的主要动作不应只有建塔，还包括：

- 放置路障
- 建立墙体
- 打开 / 关闭局部通道
- 守卫桥口、门口、坡口等关键节点
- 用技能保护工事或临时封锁路线

首版建议先只开放：

- 放置路障
- 拆除己方路障
- 建造塔

### 2. 敌人动作

敌人不再只有移动与承伤两种行为，而应至少区分为：

- `普通敌人`
  - 按当前路线前进
- `工兵`
  - 拆除阻挡自己前进的玩家工事
- `重甲单位`
  - 在玩家打开走廊后形成推进压力
- `斥候`
  - 利用已打开的短路快速突破

首版最关键的是 `工兵`，因为它直接代表敌人对路线控制权的反击。

### 3. 工兵的行为模型

建议首版工兵采用最简单可控逻辑：

1. 先按当前地图尝试正常寻路
2. 如果发现最近有效路线被工事延长太多，则寻找可攻击工事
3. 锁定一处工事并前往施工点
4. 持续攻击工事直到被击杀或工事破坏
5. 工事破坏后通知系统进行路径重算

这样设计的好处是：

- 逻辑清晰
- 玩家容易理解
- 调试成本低

## 六、核心模块拆分建议

### 1. 地图与规则层

建议新增以下模块：

- `GridMap.lua`
  - 管理地图网格、地块类型、占用状态、地块查询
- `TerrainRules.lua`
  - 定义地块通行性、成本、建造合法性和破坏规则
- `RoutePlanner.lua`
  - 负责 A* 或其他寻路实现、路径缓存、重算请求

### 2. 玩家建造层

- `Structure.lua`
  - 管理路障、墙体、门、可破坏工事等对象
- `BuildValidator.lua`
  - 校验建造位置是否合法，是否会导致完全封路
- `PlacementPreview.lua`
  - 提供预览颜色、非法提示、关键节点高亮

### 3. 敌人 AI 层

- `Enemy.lua`
  - 保留通用敌人状态、移动、伤害、死亡
- `ConstructionAI.lua`
  - 处理工兵目标选择、施工逻辑、破坏逻辑
- `ThreatEvaluator.lua`
  - 评估哪些工事最值得破坏

### 4. 调试与可视化层

- `RouteDebug.lua`
  - 画出当前路径、路径成本、瓶颈点
- `GridDebug.lua`
  - 显示地块类型和占用状态
- `EventLog.lua`
  - 输出“路径重算”“工事破坏”“非法封路”等事件

### 5. 现有模块的改造建议

- `Path.lua`
  - 短期作为兼容层存在
  - 长期应由 `GridMap.lua + RoutePlanner.lua` 替代
- `WaveManager.lua`
  - 增加工兵与破路单位配置
- `UI.lua`
  - 增加非法封路提示、路线切换提示、工事受击提示
- `Config.lua`
  - 增加地块成本、工事耐久、重算开关等参数

## 八、MVP 实施顺序

### 第一步：把地图从“画出来”升级为“可计算”

目标：

- 地图能够表示通行状态
- 可以通过网格查询敌人是否能走过去

任务：

- 从 `map.json` 或临时配置生成网格
- 给每个地块分配通行属性
- 跑通一版最短路径搜索

通过标准：

- 地图上任意阻挡变化都能重新算出路线

### 第二步：接入玩家路障

目标：

- 玩家放置路障时，路线发生变化

任务：

- 定义路障数据
- 实现放置、删除、渲染
- 放置后触发路径重算

通过标准：

- 同一关卡中玩家可以明显改变敌人的前进轨迹

### 第三步：加入非法封路校验

目标：

- 防止玩家直接把敌人完全封死

任务：

- 放置前预演一次新地图状态
- 如果无路可走，则禁止放置
- UI 给出明确提示

通过标准：

- 玩家能够理解什么情况算非法封路

### 第四步：加入工兵

目标：

- 敌人能对玩家工事做出反制

任务：

- 实现工兵目标选择
- 实现工兵攻击工事
- 工事破坏后触发路径重算

通过标准：

- 工兵出现后，玩家不能只靠堆路障无脑赢

### 第五步：做出一张主打关卡

目标：

- 验证动态路线是否真的有产品价值

任务：

- 做一张带桥口、门口或窄路瓶颈的地图
- 配置 5 到 8 波敌人
- 安排几波纯冲线和几波破路单位混编

通过标准：

- 玩家能清晰感受到路线争夺的乐趣

## 九、建造合法性规则

动态路线系统最容易做坏的地方，是让玩家封死所有路线，或者让玩家完全无法理解为什么不能放。

建议采用以下规则：

### 1. 基础校验

- 只能在 `fortifiable` 地块放置路障
- 不能与已有塔或工事重叠
- 不能放在出生点和终点

### 2. 封路校验

- 预演放置后重新寻路
- 如果从任意出生点到终点都无有效路径，则禁止放置

### 3. 特殊保护规则

建议至少保留一个：

- 核心主路不可完全封死
- 某些关键门位不可建造
- 关卡强制保留最小通行宽度

### 4. UI 提示规则

非法时建议明确提示原因：

- `此处不可施工`
- `会完全封死敌人路线`
- `与已有建筑冲突`

玩家最怕的不是限制，而是不知道为什么被限制。

## 十、敌人破路规则

### 1. 工兵优先级建议

工兵选择破坏目标时，可按以下顺序评估：

1. 是否阻挡当前最短路径
2. 破坏后是否能明显缩短路线
3. 是否位于关键瓶颈点
4. 是否离当前工兵较近

### 2. 破坏表现建议

工兵破坏工事时，建议同时提供：

- 受击闪烁
- 耐久条
- 施工音效或敲击反馈
- 即将破坏提示

### 3. 避免失控的限制

为了让系统更可控，首版建议限制：

- 每波工兵数量
- 工兵只拆 `玩家工事`，不改原始地图
- 工兵只能拆最近的一层阻挡，不做复杂多段施工

---

## 十一、UI 与反馈要求

动态路线系统的信息量明显高于普通塔防，因此 UI 必须承担解释责任。

建议至少提供：

- 当前可放置区域高亮
- 非法封路提示
- 当前敌人预计路线预览
- 工兵锁定目标提示
- 工事耐久显示
- 路线切换提示

如果路线发生了重大变化，建议加入短暂提示：

- `敌人改道`
- `工事被破坏`
- `主路已重算`

---

## 十二、数值设计建议

首版不要急着细磨所有数值，但必须保证几个关系成立：

### 1. 路障成本

- 不能便宜到能无脑铺满地图
- 不能贵到玩家舍不得用

建议首版把路障定位成：

- 比基础塔便宜
- 但不是白送

### 2. 工兵强度

- 本体不应太肉
- 但如果放着不管，会带来高战略收益

玩家应该感受到：

- `工兵本身不强，但必须优先处理`

### 3. 路线收益

玩家通过控路得到的收益，建议主要体现在：

- 增加敌人在火力区停留时间
- 让减速塔和范围塔更容易打满收益
- 给重塔争取更多输出窗口

不要只让控路带来“多走两格”的微弱收益，否则系统价值不明显。

---

## 十三、测试清单

### 1. 功能测试

- 放置路障后路线是否正确刷新
- 拆除路障后路线是否正确恢复
- 非法封路是否被阻止
- 工兵是否会锁定合理目标
- 工事被摧毁后敌群是否会重新改道

### 2. 可读性测试

- 玩家是否看得懂哪里可施工
- 玩家是否知道为什么不能放置
- 玩家是否能识别工兵正在干什么
- 玩家是否能察觉路线变化

### 3. 平衡测试

- 纯堆路障是否过强
- 工兵是否过弱或过强
- 动态路线是否真正改变了塔的价值排序

### 4. 性能测试

- 多个敌人同时重算路径时是否掉帧
- 多个工事连续被破坏时是否卡顿
- 地图调试绘制打开后是否影响帧率

---

## 十四、后续扩展方向

当首版 `路障 + 工兵` 成立后，再考虑逐步加入：

- `搭桥`
  - 用于河流、壕沟、断桥地图
- `架梯`
  - 用于城墙、高地地图
- `钻地`
  - 用于矿坑、洞穴地图
- `可切换城门`
  - 让玩家拥有短时分流能力
- `反施工塔`
  - 专门拦截工兵或保护工事
- `改造型 Boss`
  - 直接破坏关键节点或重构战场

建议一次只加一个新路线机制，并围绕它设计整张地图。

---

## 十五、开发结论

动态路线系统真正要解决的，不是“让敌人更聪明”，而是让塔防的核心玩法从：

- 守固定路

升级为：

- 抢路线控制权

对当前项目来说，最合理的推进顺序是：

1. 先保住现有原型的稳定性
2. 再重构地图与寻路底层
3. 先做 `玩家路障`
4. 再做 `工兵破路`
5. 最后用一张主打地图验证系统价值

如果这个闭环成立，后面的内容扩展、美术替换和 Boss 设计都会更有方向感。
